## High

### 1. Echo detection is still broken for most real payloads (`src/data/firebase.js`)
S79 fixed the key-order half of `isEcho()` with `stableStringify()`, but there's a second
mismatch source it doesn't cover: **Firebase RTDB strips empty arrays, empty objects, and
null values from written data.** The value `onValue` hands back omits any key the app wrote
as `[]` / `{}` — and the sync payload almost always contains some (`presence: {}`,
`activeBundles: {}`, a character's empty `conditions: []`, empty `quests`, early `ooc`, etc.).
So `stableStringify(written) !== stableStringify(echo)` on nearly every write, and the
device's own 3s-debounced sync writes still get reprocessed as remote changes. The S79
`reconcile()` fix masks most of the UI damage, but the echo check itself is doing ~nothing.

**Fix:** normalize both sides the way RTDB does — recursively drop empty arrays/objects and
null/undefined values — before stringifying in `markWritten()`/`isEcho()`. Easily unit-testable.

### 2. The S62 character union-merge does not exist in the code (`src/data/persist.js:269`)
`mergeCampaign()` is `{ ...localC, ...cloudC }` — cloud `characters` wholesale replaces
local. The workboard and session docs describe a `mergeCharacters()` union-by-id fix for the
"guest character Nyx disappeared on iOS tab-kill" bug ("committed + pushed 75c693f"), but:
- `git log --all -S mergeCharacters -- src/data/persist.js` finds **zero commits, ever**
- commit `75c693f` doesn't exist in this repo's history

(S80's session log already noticed the citation was wrong mid-implementation; this confirms
the underlying fix is genuinely absent, not just mis-cited.) Consequence: a character that
exists locally but hasn't reached the cloud yet (offline blip, write still pending) is
silently deleted the next time a newer cloud snapshot merges — and because of finding #1,
merges run on nearly every sync round-trip in multiplayer. The original S62 bug is likely
still live. Docs (workboard Known Issues, decisions.md) claim a fix the code doesn't have.

**Fix:** implement the union-by-id merge for `characters` in `mergeCampaign()` (the
`unionById` helper for messages is right there as a pattern), and correct the docs.

### 3. Provider health tracking is a silent no-op (`src/ai/providers.js:28`)
`recordFailure()` mutates the Solid store proxy directly (`health.failures++`). Solid
ignores direct store mutation — **verified live in Node against this repo's solid-js: the
mutation neither throws nor sticks; `failures` stays 0.** So `isHealthy()` always returns
true, the exponential cooldown never engages, and `pickHealthyProviders()` never reorders.
Behavioral effect: once Gemini starts failing, every single message still walks the full
Gemini retry chain (2s/4s/8s waits) before falling back to OpenRouter — every time. (Also
means S78's care to keep `callProviderOnce` from "poisoning provider health" was protecting
a mechanism that never worked.)

**Fix:** write through `setStore`/`systemSet` (e.g. `setStore('system','providers','health',provider,...)`).

### 4. Database rules: any anonymous user can read/write everything (`database.rules.json`)
All three top-level paths (`campaigns/$uid/$campaignId`, `players/$uid`, `shared`) grant
`.read`/`.write` to `auth != null` — and auth is anonymous sign-in, so *anyone* who opens
the app (or scripts against the public Firebase config, which is committed in the repo) is
"authenticated." Anyone who learns or guesses a `uid~campaignId` (invite links hand it out;
any past guest keeps it forever) can read or overwrite that campaign — and can write **any
other player's** `players/{uid}/active` / `joined` pointers, redirecting what their device
restores on boot. Family-scale threat model, granted, but this is one rules file protecting
all game data with no per-user restriction and no `.validate` at all.

**Fix (minimum):** `players/$uid` → `".write": "auth.uid === $uid"`. Better: campaign
read/write keyed to a members list written at share/join time.

## Medium

### 5. Mid-stream provider failover duplicates narration (`src/ai/providers.js:229`)
In `callProvider`, if a stream dies *after* yielding chunks (network drop mid-response —
retryable errors only throw pre-stream, but a network `TypeError` mid-read lands in the
same catch), the loop moves to the next provider and streams from scratch. `engine.js`
keeps appending to the same `fullResponse`, so the player sees the first half of the
response twice, and mechanics extraction runs over the doubled text. Fix: track whether
any chunk was yielded; if so, don't fail over (surface the error / finalize partial instead).

### 6. `zone_add_enemy` mid-combat re-sorts initiative and invalidates the turn pointer (`src/ai/mechanics.js:949`)
`currentTurn` is an **index**, and `zone_add_enemy` re-sorts the whole initiative array.
Fine at kickoff (pointer 0, rolls pending), but reinforcements added mid-fight shift the
array under the pointer — whose turn it is silently changes, and Gate 2 / the turn engine
then reason from the wrong actor. Fix: capture the current actor's name before insert,
re-find their index after sort, and restore `currentTurn` (or insert sorted without
touching earlier positions).

### 7. Roll-result submissions can be re-classified into a second roll (`src/ui/play/RollBar.jsx:475`)
The non-preSend `submitAll()` path (AI-initiated `roll_request`s) sends the results via
`sendMsg(msg, { tab: 'narrative' })` **without `skipClassifier`**. In combat the classifier
bails (`combatState.active` guard) — but the AI also self-requests rolls out of combat
(S77's Animal Handling case). There, the outgoing text — the player's optional note plus
lines like "Thorn rolled 15 for Stealth" — runs through `classifyAction()` again, and an
action verb or skill word in it can match an ACTION_PATTERN and demand a *new* roll before
the first result ever reaches the AI. One-word fix: `skipClassifier: true` on that call.

### 8. `combat_end` can revert mid-combat healing (`src/ai/mechanics.js:924`)
It copies each initiative entry's HP back onto the PC. Anything that heals a PC without
syncing the initiative copy — `long_rest`/`short_rest` mid-combat set `characters.$.hp`
directly, unlike `applyDamage` which syncs both — gets overwritten back to the stale
initiative value when combat ends. Low likelihood in real play (rests mid-combat are rare),
but the write-back direction is questionable since `applyDamage` already keeps initiative
in sync; consider dropping it or syncing rests into initiative.

## Low / cleanup

- **`src/ai/mechanics.js:94`** — `if (line.includes('|') && !line.includes('|') === false)`
  is a tautology (both operands are the same test); simplify to `if (line.includes('|'))`.
- **`src/ai/mechanics.js:993`** — the `roll-request` CustomEvent has **zero listeners**
  anywhere in `src/` (S65 rerouted the old ones to `prefill-input`). RollBar reads
  roll_requests off message mechanics instead. Dead dispatch — remove or comment why it stays.
- **`src/data/firebase.js`** — every campaign sync (3s debounce) writes the full payload to
  `localStorage` (`fb_cache:` + on failure `fb_pending:`). A long campaign is a large
  synchronous main-thread write and will eventually brush the ~5MB quota; a quota failure in
  the catch block also silently kills the `fb_pending` retry record. Worth a size guard.
- **`src/ai/mechanics.js:34` (`findPC`)** — the fuzzy clause
  `lower.startsWith(pc.name.split(' ')[0])` lets an *enemy* name resolve to a PC: enemy
  "Thorn Vine" matches PC "Thorn", so `hp: Thorn Vine=5` writes the PC's HP. The
  enemy/PC collision direction is the risky one; consider requiring exact match when the
  name also exists in `combatState.initiative` as an NPC.
- **`src/data/persist.js:152`** — the local-save dependency list watches `activeBundles`
  key count only, so a pure active-flag flip doesn't schedule a local save (cloud sync is
  handled via `setActiveBundle`; the visibilitychange flush covers most of the local gap).
  Same for `presence`. Noting for completeness.
