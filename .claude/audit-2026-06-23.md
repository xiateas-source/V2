### Top 3 divergences (most significant)

**1. The plan is inverted from reality.** The workboard is a Phase-0 starting-line document; the codebase is a deployed, playable app spanning Phases 0–7. An estimated ~80% of `[ ]` items are actually built and live. The planning files describe a project that no longer exists — this is the root cause of every other drift and the single biggest risk to velocity (work gets re-scoped or rebuilt because the doc says it's missing — exactly what happened to the backstory editor in S37).

**2. Enforcement flipped from "hard reject" to "advisory flag" — a Law 2 tension.** The plan (and Law 2, "the container is the contract") specifies that `validateMechanics()` runs 9 gates that *reject* invalid mechanics. Reality: `mechanics.js → validateMechanics()` only enforces **ownership + roll_request-PC-only** (true hard rejects). The 9 gates live in a separate `gates.js` + `drift.js`, run by `engine.js` **after** `applyMechanics()`, and only produce **non-blocking UI flags**. State is already written before a gate ever fires. This is defensible (gentler UX, fewer false-reject frustrations) but it is undocumented and in direct tension with "the AI *can't* break rules, not just *shouldn't*." It needs an explicit, recorded decision.

**3. A large body of working UI is "dark" — built but unreachable.** `App.jsx` is a minimal 4-destination router (play / cargo / journal / settings). The four-mode architecture (setup/play/reference/manage) was never wired. Confirmed orphans (code exists, no navigable path): **Treasury, Compendium, Glossary, DevTools, SessionReview, Contracts, SessionZero, ContentImport, the LevelUp wizard, and the entire AppSimple child-view entry point.** Feature value is already paid for; it just has no door.

### Secondary drift (documented here so it stops accumulating)

- **Ownership reassigned, undocumented:** `xp`, `currentSlots`, `hitDice`, `resources.$.current` moved **system → AI** in `store.js` ("runtime values") — sane (ceilings = system, depletion = AI) but architecture.md/workboard still call them system-owned.
- **Mechanic vocabulary grew 65 → ~91 keys** (wagon/ox/animal/familiar HP, zone_*, temp_hp, hit_dice_use, round_advance, sp_charge, etc.). The dispatch-table pattern absorbed it cleanly, but no doc lists the current set.
- **Undocumented subsystems** absent from the module map: `rules.js` + `rules.json` (context-aware rules injection into the prompt), `quickBuild.js` (fast PC builder), `demo.js` (demo campaign), `setupPrompts.js`, `keys.js` (API-key persistence split from firebase), `sync.js`, adventure parsers, `chunkSplitter.js`, `KeyGate`, `RollBar`, `TurnPrompt`, `SituationBar`.
- **LevelUp wizard is not event-driven** as decided ("the character IS the notification"). It exists but nothing triggers it on XP-threshold; the XP-glow→wizard flow is unwired.

## Step 2 — Strategic Plan Optimization

**Core insight:** the bottleneck is *not* feature-building velocity — it's that (a) the doc doesn't know what exists, (b) finished features are dark, and (c) the enforcement spine has quietly softened. The old Phase 4→5→6→7 "build mode-by-mode" sequence is now wrong: those modes are largely *coded*. Re-sequencing front-loads **reconciliation, exposure of built work, and integrity** ahead of net-new features. Three structural changes:

1. **Insert a doc-reconciliation pass as Phase A** before any feature work. Cheapest, highest-leverage action; prevents re-building.
2. **Insert a "wire the dark UI" Phase B** — a single navigation/mode-shell effort that lights up ~9 already-built screens. This phase did not exist in the old plan; it is now the highest ROI work available (value already built, only routing missing).
3. **Promote the enforcement-model decision (Phase C) ahead of new gates/features** to protect Law 2 integrity while the contract is still small.

Only after A/B/C do net-new features (content reachability, push/multiplayer, child view) make sense — building more dark features on a stale map would compound all three top divergences.

## Step 3 — Enhanced Project Map (drop-in for `workboard.md`)

### Phase A: Reality Reconciliation
- **Core Objective:** Make the planning docs match the deployed code so no built feature is ever rebuilt.
- **Strategic Improvement:** New phase. The old plan had no mechanism to detect that Phases 0–7 had silently shipped; S37 already lost work to a stale map. This stops the bleed before adding scope.
- **Feature Breakdown:**
  - Re-mark workboard Phases 0–3 `[x]`; mark Phases 4–7 per actual reachability (`[x]` reachable, `[~]` built-but-dark).
  - Update `architecture.md` module map to include `gates.js`, `drift.js`, `rules.js`, `messages.js`, `setupPrompts.js`, `keys.js`, `sync.js`, `quickBuild.js`, `demo.js`, content adventure parsers, `RollBar`, `TurnPrompt`, `SituationBar`, `KeyGate`, `rules.json`.
  - Correct ownership tables (`xp`/`currentSlots`/`hitDice`/`resources.current` → AI-runtime) in architecture.md + workboard state shape.
  - Record the current ~91-key mechanic set as the canonical list.
- **Dependencies:** None — pure documentation against existing code.
- **Verification Criteria:** Every `src/` module appears in the architecture module map; no workboard item marked `[ ]` that has shipped code; ownership tables match `store.js` exactly.

### Phase B: Light the Dark UI (Navigation & Mode Shell)
- **Core Objective:** Give every already-built screen a navigable route.
- **Strategic Improvement:** Net-new phase, highest ROI. The old plan would have "built" reference/manage modes that already exist in code — wasted effort. This converts sunk build cost into shipped features via routing only.
- **Feature Breakdown:**
  - Expand `App.jsx` from 4 hardcoded views into the planned mode shell (setup/play/reference/manage) without rebuilding the leaf screens.
  - Wire reference: **Compendium, Glossary, Treasury** (Treasury already linked from CharSheet Equipment footer per spec — honor tap-to-source).
  - Wire manage: **DevTools, SessionReview, Contracts** behind the Settings/manage entry.
  - Wire setup: **SessionZero, ContentImport** into the onboarding/campaign-config flow.
  - Make **LevelUp** event-driven (XP-threshold → glow → wizard), per the standing decision.
  - Decide AppSimple exposure (separate URL/entry mount) or defer to Phase F.
- **Dependencies:** Phase A (so routing targets are known); existing leaf components (no changes to their internals).
- **Verification Criteria:** From a running session, every built screen is reachable in ≤2 taps; no `import`-orphaned `.jsx` remains except an intentionally-deferred AppSimple; LevelUp fires from XP without a manual button.

### Phase C: Enforcement Integrity (Law 2 Decision)
- **Core Objective:** Make an explicit, recorded choice about which gates hard-reject vs. advise, and align code + contract + docs.
- **Strategic Improvement:** Reframes Phase 2 from "build 9 gates" (done) to "decide the gates' *authority*." Catches the silent softening of Law 2 before more features lean on the assumption that gates block.
- **Feature Breakdown:**
  - Classify each of the 9 gates: hard-reject (pre-apply, in `validateMechanics`) vs. advisory flag (post-apply, current `gates.js`).
  - Promote the high-confidence, low-false-positive gates (candidates: Gate 4 scene-transition hold, Gate 6 spell/slot validation) to pre-apply rejection; keep prose-pattern gates (1, 3, 5, 7) advisory.
  - Document the split in `decisions.md` and `enforcement-spec.md`; update Law 2's "container is the contract" examples to reflect advisory-vs-enforced reality.
  - Add the contract-shrink note: each promoted gate is one fewer prompt clause.
- **Dependencies:** Phase A (accurate gate inventory).
- **Verification Criteria:** Each gate tagged enforced/advisory in code and docs; a fabricated pre-apply violation for a "hard" gate never reaches state; `decisions.md` carries the rationale.

### Phase D: Content Pipeline Reachability & Depth
- **Core Objective:** Make the built parser stack usable by a player and close the seed-data gap.
- **Strategic Improvement:** Old Phase 7 assumed parsers were unbuilt; they exist. Work shifts from "write parsers" to "expose + harden + fill the L3+ spell gap" — a smaller, sharper scope.
- **Feature Breakdown:**
  - Surface ContentImport (Phase B routes it); add preview-before-commit to IndexedDB.
  - Episode/module tracking spec + `module_episode` wiring (still genuinely unbuilt design work).
  - Spell DB expansion L3+ / pre-built class-progression JSON downloads (seed currently stops at L2 — casters past L5 get empty pickers).
  - `bundles.js` generate/import flow validation.
- **Dependencies:** Phases A–B; `normalizer.js` schemas; IndexedDB stores.
- **Verification Criteria:** A player imports a markdown adventure end-to-end into IndexedDB; a L5+ caster opens a populated spell picker; one shared bundle round-trips between two devices.

### Phase E: Multiplayer Activation & Push
- **Core Objective:** Prove the second-device experience and add the notification layer.
- **Strategic Improvement:** `sync.js` already runs at boot, so the old "build Firebase sync" item collapses to *verification* + the genuinely-missing push layer — the one piece that makes 2-player OOC viable.
- **Feature Breakdown:**
  - Two-device sync soak test (chat merge, combat `currentTurn`, dirty-edit guard).
  - Web Push + FCM (the unbuilt half): OOC messages, others' Narrative turns, state-needs-attention, combat-turn prompts.
  - Wire push into TurnPrompt (open item from S37).
- **Dependencies:** Phases A–B; `sync.js`; PlayerOnboard identity.
- **Verification Criteria:** Two devices stay consistent through a combat round; backgrounded device receives an OOC push and a turn push.

### Phase F: Child View & Polish
- **Core Objective:** Ship the AppSimple entry point and remaining play-UI links.
- **Strategic Improvement:** Deferred to last deliberately — depends on real play observation with the child (per existing open question), and on B/C/D/E being stable so the simplified root wraps a finished engine.
- **Feature Breakdown:**
  - Mount AppSimple at its own entry; define simplification scope from actual play.
  - Citation linking + inline NPC-name linking in chat (built linking tech extends here).
  - Combat Phase 2 (visual tile map) if play demands it.
- **Dependencies:** Phases B–E; play-test feedback.
- **Verification Criteria:** Child view loads against the same Firebase/engine with larger targets; NPC names and rules citations in chat are tappable to source.

---

### Recommended build order
**A → B → C** (reconcile, expose, secure) **→ D → E → F** (extend). A/B/C are low-risk, high-leverage, and mostly non-feature work that de-risks everything after them. Do not start net-new feature scope (D+) until the map matches reality and the dark UI is lit.
